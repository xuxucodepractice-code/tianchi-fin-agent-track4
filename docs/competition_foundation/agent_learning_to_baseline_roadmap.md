# Agent Learning To Baseline Roadmap

整理日期：2026-06-23  
目标：沿用原 AI Agent 学习计划的主线，逐步过渡到理解比赛 baseline，并最终具备独立优化、修改和重构 baseline 的能力。

## 总体路线

这条路线分为三个大阶段：

1. Agent 基础学习与逐步摸索
2. 理解比赛 baseline
3. 具备完整优化和修改 baseline 的能力

核心原则不是一开始就冲 baseline，而是先把一个人如何解题、程序如何辅助、Agent 如何循环这几件事逐步摸清楚。等这些底层动作清晰后，再理解 baseline，就不会把它看成黑盒。

## 阶段一：Agent 基础学习与逐步摸索

对应原学习计划的阶段 00 到阶段 04。

### 目标

建立 AI Agent 的基础直觉：

- Agent 本质上不是魔法，而是循环。
- 工具调用不是模型自己执行工具，而是模型提出结构化请求，程序负责执行。
- 记忆、工具、状态、评估和 guardrails 是逐步长出来的工程能力。
- 高代价、可验证任务要优先 workflow；开放任务才适合交给 agent 自主决策。

### 阶段 00：人肉理解任务，再手搓最小循环

先不碰复杂 baseline。

要做的事：

- 人工解 1 道 A 榜题。
- 理解题目、`doc_ids`、原文证据、选项判断、答案格式。
- 手写最小 `Think -> Act -> Observe` 循环。
- 工具只保留最小集合，例如计算器、读取文本片段。
- 不使用 LangChain / LangGraph。

通关标准：

- 能解释一个人解题时每一步在做什么。
- 能写出一个零框架的小循环，让模型在给定证据下完成一次判断。

对应理论：

- ReAct: https://arxiv.org/abs/2210.03629

### 阶段 01：真实工具与最简记忆

要做的事：

- 接入真实工具调用。
- 给工具定义清晰 schema。
- 接入文档读取、关键词搜索、Token 统计等基础工具。
- 保存历史 messages。
- 让模型基于证据判断单选、多选、判断题。

通关标准：

- 能跑通 1 道题的半自动流程：

```text
题目 -> doc_ids -> 读取原文 -> 检索证据 -> Qwen 判断 -> 答案校验
```

对应理论：

- Anthropic: Building Effective Agents

### 阶段 02：可观测状态图

在理解底层循环后，再上框架。

要做的事：

- 把流程拆成显式节点：

```text
load_question -> map_doc_ids -> parse_documents -> retrieve_evidence -> reason_options -> normalize_answer -> record_tokens
```

- 学习 routing、orchestrator-workers、evaluator-optimizer 等模式。
- 接入 tracing 或日志，让每一步可观察。

通关标准：

- 能通过 trace 或日志定位为什么答错：
  - 找错文档
  - 检索不到证据
  - 证据足够但推理错
  - 答案格式错
  - Token 太高

可选课程：

- Hugging Face AI Agents Course
- DeepLearning.AI Building Code Agents with smolagents
- 吴恩达 Agentic AI Course

### 阶段 03：评估与可靠性

要做的事：

- 用 A 榜 100 题建立评估 harness。
- 自动统计可验证指标：
  - 提交格式合法率
  - 每题 Token 消耗
  - 每领域平均 Token
  - 人工抽样准确率
  - 线上反馈后的错题类型

- 加入确定性 guardrails：
  - 单选只能输出 A/B/C/D 之一
  - 多选必须排序、去重、无分隔符
  - 判断题只能输出 A/B
  - `summary` 行必须正确统计 Token

通关标准：

- 能用数字描述系统表现。
- 能把失败 case 分类，并提出下一轮优化方向。

对应理论：

- Lilian Weng: LLM Powered Autonomous Agents

### 阶段 04：比赛版 Agentic RAG

这一步才进入更完整的 Agentic RAG。

注意：本比赛禁止 embedding 模型参与检索和推理，所以这里的 RAG 不是传统向量 RAG。

重点方向：

- 规则检索
- BM25 或关键词检索
- 标题、页码、章节、条款编号索引
- 表格字段索引
- 金额、时间、比例、实体抽取
- Qwen 辅助的查询改写和证据判断
- 多轮证据补充
- 证据压缩

通关标准：

- Agent 不只是能回答，还能给出可追踪证据链。

## 阶段二：理解比赛 baseline

进入 baseline 的前提是：已经知道人怎么解题，也知道程序如何把动作串起来。

### 目标

把 baseline 拆成可理解的模块，而不是把它当成一整块代码。

### 需要理解的模块

| 模块 | 要理解的问题 |
| --- | --- |
| 数据读取 | 题目 JSON 如何加载，字段含义是什么 |
| doc_id 映射 | `doc_ids` 如何对应 PDF/HTML/TXT |
| 文档解析 | PDF、HTML、TXT 分别怎么抽取文本 |
| chunk 切分 | 为什么这样切，切太长或太短有什么问题 |
| 检索 | 如何不用 embedding 找到候选证据 |
| prompt | 如何把题目、选项、证据组织给 Qwen |
| 推理 | 单选、多选、判断分别怎么判断 |
| 答案校验 | 如何保证输出符合提交格式 |
| Token 统计 | 哪些调用要计入 Token，如何汇总 |
| 日志与证据 | 如何记录每道题的证据和失败原因 |
| 提交生成 | 如何生成 `answer.csv` 和 `evidence.json` |

### 学习方式

不要一次读完整 baseline。

按这条路径拆：

1. 跑通 1 道题。
2. 只读这 1 道题经过的代码路径。
3. 画出数据流。
4. 改一个小参数，看结果变化。
5. 再扩展到 5 道题。
6. 最后扩展到 100 道题。

### 通关标准

能不用看代码，口头解释 baseline 的完整流程：

```text
questions JSON
-> doc_id map
-> raw document
-> parsed text
-> chunks
-> retrieval
-> Qwen reasoning
-> answer normalization
-> token summary
-> answer.csv
```

并且能指出每一步可能失败在哪里。

## 阶段三：具备完整优化或修改 baseline 的能力

### 目标

从“能跑 baseline”升级为“能判断哪里值得改，并能验证改动是否有效”。

### 优化方向

#### 文档解析优化

- PDF 保留页码、标题、表格和条款编号。
- HTML 去噪，保留正文、发布时间、附件关系。
- TXT 法规文本按章、节、条切分。
- 监管领域单独处理 HTML、附件 PDF、strict TXT 三种来源。

#### 检索优化

- 为每个领域设计不同检索策略。
- 财报重点检索指标、年份、公司名、现金流、研发、分红。
- 保险重点检索责任、领取、身故、退保、年龄、比例。
- 合同重点检索发行人、评级、期限、利率、受托管理人、违约条款。
- 监管重点检索法规名称、条号、施行日期、义务主体、期限。
- 研报重点检索行业、公司、指标、预测值、同比环比。

#### 推理优化

- 按选项逐个判断，而不是让模型一次给总答案。
- 要求模型输出支持、反驳或证据不足。
- 多选题先判断每个选项真假，再排序拼接答案。
- 对数值题加入确定性计算。

#### Token 优化

- 避免整篇长文档输入。
- 只把候选证据和必要上下文送入模型。
- 对多轮检索结果做短证据压缩。
- 记录每题 Token，找出高成本题型。

#### 可靠性优化

- 输出必须经过格式校验。
- 缺证据时允许重试检索。
- 证据冲突时触发二次判断。
- 每道题保存 `evidence.json`，方便复盘。

### 每次优化必须回答的问题

- 改了哪个模块？
- 为什么认为它会提高分数？
- 对准确率有什么影响？
- 对 Token 有什么影响？
- 哪类题改善了？
- 哪类题变差了？
- 是否增加了复杂度和维护成本？

### 最终通关标准

达到以下状态，才算真正具备修改 baseline 的能力：

- 能独立跑通 A 榜 100 题。
- 能解释每个模块的职责和失败模式。
- 能根据错题定位是解析、检索、推理还是格式问题。
- 能提出优化方案并实现。
- 能用评估结果证明优化有效或无效。
- 能生成合规的 `answer.csv`、`evidence.json` 和复现说明。

## 推荐执行顺序

```text
人肉解 1 题
-> 工具辅助解 1 题
-> 模型基于证据判断 1 题
-> 自动跑 1 题
-> 自动跑 5 题
-> 自动跑 100 题
-> 读懂 baseline
-> 复现 baseline
-> 逐模块优化 baseline
```

这条路线的重点是循序渐进。baseline 不是起点，而是中段目标；最终目标是你能亲手拆开它、解释它、改造它。
