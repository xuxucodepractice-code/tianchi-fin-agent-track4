### v14 —— 2026-07-09（首个 agent 推理循环：Self-RAG 自检 + CRAG 定向补查）
- 脚本：`submissions/run_v14_agent_loop.py`
- 系统：**路线从"调压缩prompt"转向"给agent加推理循环"**——这是第一个有真推理的版本。在 v13b(question-aware压缩)基础上，答完不再收工，而是进入循环：**答→自检(审证据缺口)→若缺则BM25定向补查→补进笔记→再答**，最多补 MAX_ROUNDS=2 轮，自检说OK或补不到新证据即停。
  - 自检(build_selfcheck_prompt)：纯prompt让模型审自己的初答，输出严格JSON `{"ok":bool,"missing":[缺口词]}`。**多选查漏选**("未选选项是否因笔记没提它=证据被漏")、**tf查复合句**("且/而连接的每个子事实都有证据吗")、mcq查区分度。
  - 定向补查(supplementary_retrieve)：把缺口关键词在**本题给定文档内**用BM25再检索SUPP_TOP_N=6段，补进笔记。**纯关键词、无embedding、无小模型、不联网**。
- 理论支撑(改进策略.md 第14节)：Self-RAG(arXiv:2310.11511, 边答边自检relevance/support、retrieve-on-demand) + CRAG(arXiv:2401.15884, 评估器判信心触发纠错补查)。合规：不训练模型、不用小模型、不联网、无embedding——纯prompt自检 + BM25补查。
- 相对基线：相对 v13b **只加"自检→补查→修正"循环这一层**，压缩/答题/检索全复用 → 干净归因agent循环的作用
- 结果：**0空答案**。本地对Claude尺子：全部 **70%**(multi **71%** / tf **75%** / mcq 60%)。**全程触发20轮补查**(15题至少补1轮)，agent确实在"发现证据不足→回去补"
- 过拟合测试(每版必做)：练习堆70.0% / 封存堆70.0% / **分差+0.0点 ✅ 全场最健康**——两堆分毫不差，说明能力**完全不依赖调这100题**，是agent循环带来的真泛化。对"B榜换新题必须也好"这是最理想读数
- Token：total **1,824,731**(in 1,615,912 + out 208,819)，TokenScore≈**0.6351**，系数≈0.8905。**token涨到182万**(多了自检+补查调用)是本版最大代价
- 结果：**63.2266**(本地估62.34，尺子差<1分再次验证准)；反推准确率≈63.23/0.8905≈**71%**，与本地70%一致
- **归因拆解(v10直接喂70.91 vs v14 63.23，差7.68分)**：按乘法公式拆——
  - **准确率掉(77%→71%)吃掉5.34分**((0.77-0.71)×0.891×100)≈**占70%，主因**
  - **token涨(132万→182万)吃掉2.34分**(0.77×(0.921-0.891)×100)≈占30%，帮凶
  - **关键**：v14准确率71%不但没超v10(77%)，反而低6点。agent循环把压缩的坑补回了一部分(v13c 68%→v14 70%)，但**它是套在"压缩"这个已掉到68%的地基上修补**，补到70-71%就到头，追不回v10不压缩的77%
- 一句话结论：**agent循环方向对(泛化+0.0最健康、治病有效：tf 70→75、multi 69→71)，但套在次优的压缩地基上，追不上最简单的v10直接喂(77%)。下一步 v15：把v14的自检+补查循环搬到v10直接喂(76%)的高地基上(去掉压缩层)——同时治两刀(准确率地基68→77、token去掉压缩层的开销)，验证agent循环加在最好地基上能否突破76%。**

- 结果：**0空答案**。本地对Claude尺子：全部**73%**(multi 72% / tf **85%** / mcq 60%)。触发补查**38轮**
- 过拟合测试(每版必做)：练习堆73.3% / 封存堆72.5% / 分差**+0.8点 ✅健康**
- Token：total **3,698,432**(in 3,516,022 + out 182,410)，TokenScore≈**0.2603**，系数≈0.7781。**token爆到370万**是本版致命伤——比v14(182万)翻倍、比v10(132万)近3倍
- **好消息(方向证实)**：去压缩层后准确率70%→**73%**，尤其**tf 75%→85%**(复合句漏半句的病治好一大截)；过拟合+0.8泛化健康；单看准确率v15(73%)是所有"有循环"版本最高
- **坏消息(token爆炸)**：估分**56.80**，反比v10(69.99)**低13分**。根因：**自检和补查都在"直接喂"地基上重复喂检索段**——v14每轮喂几百字压缩笔记(便宜)，v15每轮喂完整检索段(multi top24≈上万字)，38轮补查×每轮上万字→token滚上去。**"循环重复喂大证据"这个交互效应，把准确率的收益全吃光还倒亏**
- 分数账(公式拆解，回答"到底准确率还是token问题")：`分=100×准确率×(0.7+0.3×TokenScore)`。**token那项被封顶(0.7~1.0)、只剩8%空间；准确率线性直乘、无上限**。反证：就算token压到0(TokenScore=1)，v15的73%准确率也只能到73分。**结论：现阶段唯一杠杆是准确率**——但v15的教训是"循环把大证据重复喂"会让token从次要变主导
- 一系列零成本验证(定位省token方案，均用已跑数据+现成代码，不烧token)：
  1. **BM25置信度门控**(probe_bm25_confidence)：想用"检索最高分"判断哪些题跳过循环省token。**否**——高/低批准确率仅差+10点、三分位不单调(中批最低)、mcq上信号反了(答错的BM25分反更高)
  2. **按题型砍mcq循环**(probe_mcq_loop_effect，重跑15道mcq)：mcq循环13/15空转，本以为可砍。**否**——2题改了答案里1题是**救回的**(ins_a_003 初错A→补查2轮→终对D)、0题改坏，**循环对mcq净有益不能砍**
  3. **BM25反向识别漏选**(probe_omission_bm25)：想用"未选选项的BM25分"客观找漏选替代模型自报。**否**——漏选vs真不选均分仅差+22、大面积重叠，各阈值精确率卡在15~22%(抓5个真漏选只1个、4个误伤)
  4. **logprobs捕捉漏选犹豫**(probe_multi_logprobs，Self-RAG/FLARE/DRAGIN思路)：先测得qwen-plus-2025-07-28支持logprobs(判断题答'A'概率1.0，单字母无区分度)；再测10道已知漏选multi题→**意外反转**：其中6道换qwen-plus-2025-07-28**直接答对**(补上漏字母、概率≈1.0)。说明v15漏选**很大程度是网关deepseek模型弱造成的**，不全是结构病。真硬漏选(fc_a_004/ins_a_009/fin_a_012)漏字母概率极低(0.03~0.22)——**低概率=真该补查、高概率=本来就会选**，logprobs方向有效但那模型额度不够跑全量
- **文献支撑(省token方向)**：Adaptive-RAG(arXiv:2403.14403，按题难度路由：简单题不检索/中等单步/复杂才迭代) + FLARE(arXiv:2305.06983，用token低置信度触发按需检索) + DRAGIN(arXiv:2403.10081，RIND实时信息需求检测+QFS查询构造)。核心思想=**循环不该默认全开、该按需触发**。合规红线：三者原版都用小模型/训练/logprobs，我们只借"按需"思想
- 一句话结论：**去压缩层证明有效(73%、tf治到85%、泛化+0.0)，但"循环重复喂大证据"导致token爆炸370万倒亏13分。省token的4个客观门控信号全被数据否掉(BM25弱、logprobs受限)——绕一圈发现"模型自检自报"可能已是合规工具箱里最合理的怀疑点判断。真正翻盘点：实测qwen3.7-plus(v10那个77%强模型)现可调用→下一步 v16：把v15循环原样搬到qwen3.7-plus(强模型+合规可提交)，先拿干净基线分，再做增量喂省token。**

### v16 —— 2026-07-14（强模型+循环：准确率冲到77%全场最高，但token超500万预算触底）
- 脚本：`submissions/run_v16_loop_qwen37.py`（带**断点续跑**：读已有CSV非空答案跳过，防欠费中断后从头烧余额）

  - 比v15(370万)还高：**qwen3.7-plus的input量比网关deepseek更大**，循环重复喂全段(初答+自检+补查再答，每次multi上万字)把input顶到522万。**循环重复喂大证据的硬伤，在贵模型上更致命**

### v17 —— 2026-07-14（增量喂：token砍74%到136万，本地保住77%，但实测只有68.8705——首次暴露"本地估分虚高"）
- 脚本：`submissions/run_v17_incremental.py`（带断点续跑，同v16）
- 系统：**v16 + 增量喂**。相对v16唯一变量=喂给模型的文本量（方法/模型/循环决策/补查逻辑全不变，只改"每步喂多少字"），两刀：
  - **刀① 答题+自检合并成一次调用**（`build_answer_selfcheck_prompt`）：喂全段证据**仅此一次** → 同时输出`{answer, missing}`。v16是初答喂全段+自检又喂全段=两次，这里砍成一次
  - **刀② 补查再答只喂增量**（`build_reanswer_incremental_prompt`）：只喂"上一轮答案 + 新补的那几段"，**不重喂旧全段**。v16再答喂"全段+新段"，v17只喂"结论+新段"
  - `parse_ans_missing`解析合并JSON，失败退回`v6.normalize_answer`+missing空(不补查)
- 结果：**0空答案**。本地对Claude尺子**全部77%**(tf **95%!** / multi 77% / mcq 53%)。触发补查**8轮**（比v15的38轮/v16少很多——合并自检后模型更少报缺口）
- 过拟合测试(每版必做)：练习堆约77% / 封存堆71.17% / 分差健康
- Token：total **1,362,120**(136万，比v16的522万**砍74%**)，TokenScore≈**0.7276**，系数≈**0.9183**。**增量喂彻底解决token爆炸——两刀理论零准确率风险(不改任何循环决策)**
- 本地估分**70.71**(封存堆71.17)，本地看是首个可能突破v10(70.91)的合规版本
- **★致命发现：实测在线分只有 68.8705**，比本地估分70.71**低约2分**，也低于v10实测70.91。反推真实准确率≈**75%**(本地77%偏乐观约2点)
  - 归因：本地"Claude尺子"(reference_claude.csv)不是官方答案，本地77%里有约2点是"跟Claude一致但其实错"的虚高。**这是本session最重要的方法论教训：本地估分系统性偏乐观~2点，以后所有版本的本地分都要打2点折扣看**
- **诚实存疑(不敢下的结论)**：v17(75%实测) vs v10(77%实测/70.91) 差2点，但**这2点不能归因于"循环伤准确率"**：(1) v17相对v10改了**两个变量**(加循环 + 合并自检prompt)，不是单变量；(2) 2点在尺子噪声内；(3) **v16纯循环从未上线**，"强模型+循环"的真实在线效果**未验证**。所以"循环在强模型上伤准确率"目前是**未证实的猜测**，不能当结论
- 一句话结论：**增量喂工程上成功(token 522万→136万砍74%、保住本地77%)，这正是本比赛"动态记忆压缩"要考的能力。但实测68.8705暴露两件事：①本地估分虚高~2点(尺子非官方答案)；②我们的agent(循环)在强模型上没能超过最简单的v10直接喂——问题不在"该不该有循环"(比赛要求必须是agent)，而在"我们的agent还不够好"：循环的"再答"有时把对的初答改错(强模型过度自我怀疑)。下一步方向=让agent动作更保守(多选再答只在新证据支持下增选、绝不删改初选)，把循环从"净持平/微亏"做成"净收益"。**

### v18 —— 2026-07-15（路线A·保守动作：把"病=多选漏选"钉进动作空间→真实准确率冲到79%全场最高，token又爆炸拖低实测分）
- 脚本：`submissions/run_v18_conservative.py`（deepseek网关，同v9/v14/v15便于对照）
- 系统：**相对v15唯一改动=多选题的"再答"只允许基于新证据补选、绝不删改初答已选的**。实现=`merge_multi_conservative(prev, new)` 取【初答已选 ∪ 再答新选】的并集 → 循环在结构上**只可能补漏、不可能把对的改错**。把v17诊断出的"病=漏选"这条领域先验直接编进agent动作空间，而非靠模型自觉

- 结果：本地对Claude尺子**全部77%**(tf 85% / multi **77%** / mcq 67%)。触发补查38轮。1空答案(ins_a_006网关推理截断返回空，偶发抖动非逻辑bug)

- Token：total **3,652,409**(365万)，TokenScore≈**0.2695**，系数≈0.7809。**又是v15/v16同款病：网关直接喂全段重复喂→token爆炸**

- **★关键读数(把token项剥离看准确率)：实测61.6876 ÷ 系数0.7809 = 真实准确率≈79%——所有合规提交里最高**，超过v10天花板77%、比v17(75%)高4点
  - **路线A验证成功**：把"多选只增不减"钉进动作空间后，真实准确率从v17的75%抬到**79%**，正好印证v17的诊断——之前掉分就是循环的"再答"把对的初选删改错了，结构上禁止删改后血止住了
- **v18分数低纯粹是token问题**，与准确率无关：365万token把系数压到0.78。这与v15/v16完全同病，解药已知=v17的增量喂
- 一句话结论：**路线A(保守动作)是准确率的干净胜利——真实准确率79%全场最高、超过v10天花板，证明"把领域知识(病=漏选)编进agent动作空间"这条路对了。v18实测分低只因走的是网关直接喂(token 365万)，不是准确率问题。下一步 v19：v18保守动作 + v17增量喂 + qwen3.7-plus = 79%准确率 + 136万量级token → 理论分≈79%×0.918≈72.5，首个可能突破v10(70.91)的合规版本。**
