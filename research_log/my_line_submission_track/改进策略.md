# 改进策略 — 金融长文本 Agent 提分路线图

> 配套文件：`afacClaude.md`（工作手册）、`thinking.md`（思考记录）、`result.md`（结果台账）。
> 本文件用途：系统记录这个赛题（动态记忆压缩与高效问答）的**常用方法、能不能用、改进优先级**，作为后续每一版优化的选菜单。
> 建立日期：2026-07-04（v3 首次提交后，得分 29.2375，准确率≈30%）

---

## 〇、先记住比赛的本质（所有方法都围绕它）

> **文档很长（几百页）塞不进模型，还要省 token。**
> 核心矛盾 = "只喂关键的那几段" vs "别丢了证据"。
> 谁能用最少的字喂中要害，谁赢。

两条特殊红线（决定很多通用方法在这里不能用）：
1. **禁止 embedding** 模型做检索/rerank/召回/推理 → 只能关键词/BM25/规则/条款号。
2. **要省 token**（评分 30% 看 token 效率，预算 500 万）。
3. 正式答题只能用 **Qwen 系列**；预处理（PDF/OCR）可用非 Qwen 工具，但其语义结果不得延伸到正式答题。

---

## 一、检索 / 定位类（最重要，当前最缺）

目标：从几百页里**找出该读的那几段**，而不是盲截断读开头。

| 方法 | 是什么 | 能用吗 |
|---|---|---|
| 关键词检索 | 从题目/选项抽关键词（如"保单贷款"），去文档找含这些词的段落 | ✅ 首选 |
| BM25 | 关键词加强版，按词频+稀有度打分排序取最相关段落 | ✅ 经典且强，非 embedding |
| 章节/条款号定位 | 金融文档结构强（第X条/第X章），按编号精准跳转 | ✅ 监管/合同特别有效 |
| 正则/规则抽取 | 用规则抓金额、日期、比例、期限等结构化信息 | ✅ 财报数值题有用 |
| 向量检索（embedding RAG） | 文字转向量按语义相似度找 | ❌ 明确禁止 |
| Qwen 辅助查询改写 | 让 Qwen 把题目改写成更好的搜索词再检索 | ✅ 用的是 Qwen，合规 |

**主路线：关键词/BM25/条款号找到相关段落 → 只把这些段落喂 Qwen。** 相对盲截断最大的提分点。

---

## 二、压缩 / 记忆类（赛题名"动态记忆压缩"就指这个）

目标：**有限 token 里塞进最多有效信息**。

| 方法 | 是什么 | 能用吗 |
|---|---|---|
| 抽取式压缩 | 只保留检索到的关键句/段，扔无关部分（不改写原文） | ✅ 最安全 |
| Qwen 做摘要压缩 | 让 Qwen 把长段压成短摘要再答题 | ⚠️ 能，但必须用 Qwen 做 |
| 分块+逐块处理 | 文档切块，一块块读、逐步积累结论（Map-Reduce） | ✅ 大文档常用 |
| 去重/去模板 | 年报里重复的免责声明、页眉页脚先删 | ✅ 省 token 立竿见影 |

⚠️ **合规陷阱**：规则禁止"直接使用非 Qwen 模型生成的摘要参与答题"。所以压缩/摘要**必须用 Qwen**，不能用别的模型/工具的语义摘要。抽取式（只挑原文句子、不做语义改写）最安全。

---

## 三、Prompt / 推理类（改措辞不动架构，性价比高）

目标：**同样的材料，让 Qwen 答得更准**。

- **思维链 (CoT)**：让 Qwen"先分析每个选项、找依据，再给答案"，而不是直接蹦字母。
- **逐项判断**：多选题一个选项一个选项判断"对/错/证据不足"，最后汇总（官方样例 reg_a_014 就是这么解的）。
- **证据先行**：要求"先引用文档原文，再基于原文判断"，减少瞎编。
- **Few-shot 示例**：给 1-2 个"题目+正确解法"范例，它照着学。
- **领域定制 prompt**：保险/财报/监管各写一套（解题逻辑不同）。

这类改动改几句话就能试，最适合快速迭代。

---

## 四、答案后处理类（保证不因格式丢分）

- 字母抽取、去重、排序（v3 已做 `normalize_answer`）。
- **空答案兜底**（v3 有 3 题空 = 白丢 3 分，必须处理）。
- 判断题 A/B 含义对齐（有的题 A=对、有的 A=错，看选项文字）。

---

## 五、评估 / 工程类（知道自己错在哪）

- 分领域看错题（`thinking.md` 在做）。
- 日志/trace：每题存"检索到啥、喂了啥、答了啥"，错了能倒查。
- token 预算管理：给每题设 token 上限，别让个别题吃掉预算。

## 六、当前诊断（基于 v3 = 29.2375 / 准确率≈30%）

- 官方 baseline A 榜 49%，我们 ≈30%，**低于 baseline**。
- 原因明确：**盲截断只读前 3000 字**，比 baseline 塞全文丢的证据还多。
- 但 **token 只用了 8%**（42万/500万）→ 富余极大，完全可以"多读、读对"而几乎不损 token 分。
- **结论：分数低几乎全卡在"证据没喂够"，这就是主攻方向。**

## 七、改进优先级（按提分/投入比排）

```
第1档（马上做，改动小收益大）:
  ① 修 3 题空答案 + 加思维链/逐项判断 prompt   ← 不动架构，只改 prompt
  ② 大文档去掉页眉页脚/免责声明等垃圾           ← 省 token

第2档（核心提分，改动中等）:
  ③ 关键词/BM25 检索：盲截断"读开头" → "找到含关键词的段落再读"
     ← 从 baseline 级迈向真正 RAG 的关键一步，尤其救 fc/fin/research 大文档

第3档（锦上添花）:
  ④ 分领域定制 prompt、条款号精准定位、Qwen 查询改写
```

## 八、两条可选执行路线（下一步二选一）

- **路线 A（最小改动快速验证）**：`CHUNK_CHARS` 从 3000 调大（如 10000）+ 加思维链 prompt + 修空答案。
  - 目的：验证"多喂证据就涨分"这个假设，今天就能再交一版看分。
- **路线 B（一步到位做检索）**：直接上关键词/BM25，找对段落再喂。
  - 目的：直接治本，但改动大。

**建议先 A 后 B**：小步改→交→看分→再改，符合双线循环节奏。

## 九、迭代纪律（每次改进都遵守）

```
改一个点 → 重跑 → 记 result.md（分数）+ thinking.md（为什么/下一步）
   → 提交看真分 → 对比上一版涨没涨 → 决定保留或回滚
```

- 一次只改一个变量，否则分数变了不知道是谁的功劳。
- 每个改进都要能回答："token 变化？准确率变化？哪类题变好/变差？值得保留吗？"
- 真分数说了算，不靠"我觉得变好了"。

### 10.3 现在走到菜单哪一步（下一层的病）

病根（证据没喂够）已治好。63 分反推准确率 ≈68%（还错 ~32 题），token 只用了预算 **23%**，所以现在是**"用准确率换分"阶段，不是省 token**。新暴露的问题对应本文档第 3 档"分领域定制 prompt"，已具体化为：

1. **tf 判断题结构性失效**：20 道判断题**答案全是 A**（"正确/错误"是停用词，A/B 拿到相同证据 + fallback 恒 A）。预计丢 8-10 题 —— **性价比最高的一改**。
2. **多实体/计算题证据组装失败**：保险 mcq（ins_a_001/002 型）涉及多产品各自数值，单轮 top-5 给不全 → 全 insufficient 纯猜。对应"证据组织"（相邻块合并、按产品分桶），检索层还没做。

4. **mcq fallback 纯猜**：mcq 答案 A 占 11/15，6 题 0-support 硬猜。应改"四选一比较"。

> 详细诊断见仓库 `workspace/03_baseline_improvement/v1_architecture_review.md`（队友的 v1 架构审查）。

## 十二、路线转向：从"刷检索分"到"做真正的Agent（推理+压缩）"（2026-07-08）

> 触发：v10(70.91) 后停下来反思两个更本质的问题——(1) 我的代码算不算"有推理能力的agent"？(2) 我这些分数是不是过拟合了？结论促成路线转向。本节记路线判断与新增的评测工具；分数见 result.md，个人分析见 thinking.md。

### 12.1 诊断一：现在的代码几乎没有"agent推理"，只是RAG管道

比赛目标是**做agent**，agent的核心是推理能力。但现有代码（`run_v6_retrieval.py` 那套）是**单步、直线、无反馈**的：

```
读题 → 检索(固定BM25) → 拼prompt → 问模型一次 → 抽字母 → 完
```

对照"有推理能力的agent"该有的要素，全都缺：

| Agent推理要素 | 现状 | 缺在哪 |
|---|---|---|
| 多步/迭代 | ❌ 只一步 | 不会"证据不够→再查一次" |
| 自我反思/校验 | ❌ 无 | 答完就收，不自检"答案和证据矛盾吗/多选漏了吗" |
| 动态决策 | ❌ 全写死 | TOP_N按题型硬编码，不是agent自己判断难易 |
| 工具调用循环 | ❌ 无 | 检索是硬调用，不是模型自己决定要检索 |
| 规划 | ❌ 无 | 不拆解问题 |

**关键认知**：推理目前全靠模型内部（尤其开思考时），代码本身不推理。之前想做的 v11"开思考"本质是"借模型的推理补代码没有的推理"——那是模型在推理，不是agent在推理。真正要做的是**给代码加推理循环**（如"检索→答→自检→修正"多轮），这是架构级改动，不是调参。队友那版"逐选项 support/refute/insufficient 判断"比单步管道更接近推理，可借鉴。

### 12.2 诊断二：比赛真正的核心是"压缩"，不是"检索"

赛题全名=「金融长文本 Agent **动态记忆压缩** 与高效问答」，**压缩写在标题里**。打分公式 `100 × 准确率 × (0.7 + 0.3×TokenScore)` 里 TokenScore 直接奖励少用token。

**个人判断（用户提出）**：Qwen 参数量不大、喂不下超长金融文本 → **谁能把长文压得又短又不丢关键信息，谁就赢**。此前一路"多喂证据(top8→24)"其实在往**反方向**走（token越用越多，v6的66万→v9/v10的132万）。压缩要把这条重新掰正：**目标从"多喂"转为"喂得准而少"**。

### 12.3 诊断三：过拟合——建了自己的评测台来量

- **风险**：只有这100题，v3~v11 全在同一批题上反复调参看错题，**调参过程本身在悄悄过拟合这100题**。真实泛化要用"这100题之外的新题"测，但比赛没给独立测试集。
- **对策=自建本地评测台 `my_benchmark.py`**（详见 result.md 工具注记）：
  1. **估分**：套官方公式，不上传就估线上分。已校准极准（v10估69.99/真70.91、v6估64.34/真61.46）。
  2. **看过拟合**：把100题固定分【练习堆60 / 封存堆40】(SEED固定、按题型分层)。**调参只在练习堆做，封存堆当模拟考平时不看**，两堆分差=过拟合仪表。实测 v10 分差-2.5、v6 分差-0.8，**均健康、非背题**。
  3. **诚实边界**：100题过去都看过，故此结论对"过去版本"是回溯性的；但从现在起做agent/压缩，只要不看封存那40题，它就是干净的模拟考。
- **评测台顺带暴露的短板**：mcq单选 60%（最弱题型）、fin金融领域 60%（最弱领域）——做完agent推理后可在**练习堆上**针对性看，别碰封存堆。

### 12.4 下一步（新阶段的纪律）

1. **停止对着个别错题抠TOP_N**（那是过拟合行为，top24已够）。
2. 开发阶段**先用 deepseek 网关**验证方法（用户暂无Qwen额度），所有东西完善后再切回 Qwen 并考量上下文/额度问题。
3. **主攻两件事**：① 给代码加**推理循环**（检索→答→自检→修正），让它成为真agent；② 做**压缩**（把长文压短且不丢关键信息，直接提 TokenScore）。
4. 每版都过 `my_benchmark.py`：看封存堆估分（更诚实）+ 过拟合仪表，再决定去留。

## 十三、压缩方向的参考文献与"怎么压才不丢证据"（2026-07-08）

> 背景：v12 压缩Agent（生成式重写压缩）准确率从 v10 的76%退到70%，multi掉10点。诊断出**9道多选被压缩弄错、全是漏选**——检索段和v9相同（v9同款段能答对），是**压缩这一层把多选分散的证据点删掉了**。查了两篇权威参考，找到根因的学术解释与改法。

### 13.1 参考文献

**① RECOMP（论文，arXiv:2310.04408，《Compressing Retrieved Documents for RAG》）** — 直接对口
- 把"压缩检索内容"分两类：
  - **抽取式(extractive)**：从原文**挑出**最有用的**句子**，不改写。
  - **生成式(abstractive)**：让模型**重写**成摘要（会合成、会删）。← **v12 用的就是这种，会丢证据**
- **最关键结论（命中v12的根因）**：压缩要"**为最终答题效果**训练/优化，**不是只为变短**"（optimize for end-task performance, not just to shorten）。
  - **v12的错=目标函数错了**：prompt让模型"尽量简短"→在优化"短"，模型为了短把多选第3个选项的证据删了。应改成优化"答对"。
- 另一条：无关内容可输出空串（selective augmentation），压到6%仍minimal loss——**证明压缩本身可行，关键在怎么压**。

**② LLMLingua / LongLLMLingua（微软开源，GitHub，MIT）** — 长文压缩工具
- **question-aware（问题感知）压缩**：压缩时始终盯住问题，保住"支持答案的内容"（LongLLMLingua 的 `condition_in_question`）。
- **coarse-to-fine（粗到细）**：先粗筛相关段（=我们的检索）→再细压token。报告"**只用1/4 token，RAG效果反升21%**"——压对了能又省又准。

### 13.2 ⚠️ 合规红线：参考思想可借，库不能直接用

比赛规则禁 embedding、禁非Qwen模型参与正式答题。而：
- **LLMLingua 用小模型(GPT2/BERT)算token重要性 → 违规（非Qwen）**
- 抽取式压缩若用 embedding 算句子相关性 → **违规（禁embedding）**

**结论**：不能直接装这些库，但可**借它们的思想、用合规方式（纯Qwen挑句、无embedding）实现**。

### 13.3 落地改法（v13 方向）：抽取式 + question-aware 压缩

把 v12 的"生成式重写"（丢证据）改成**抽取式 + 盯选项 + 目标改对**：
- 压缩prompt改为：**"从下列段落中，逐字摘录所有与问题各选项相关的句子（尤其含数字/条款/结论的），宁多勿漏，不要改写、不要总结。"**
- 三篇精华合一：抽取式（不改写→不丢证据）+ question-aware（盯着选项）+ 目标从"短"改成"答对/宁多勿漏"（RECOMP的核心教训）。
- 合规：纯 Qwen 挑句，无 embedding、无小模型。
- 单变量：只改压缩prompt，检索/答题不动；跑完过 `my_benchmark.py` 做过拟合测试。
- 预期：多选笔记变长（token增，现阶段先不管token），多选漏选缓解、准确率回升。

## 十四、路线再定位：从"调压缩prompt"到"给agent加推理循环"（2026-07-09）

> 触发：v13 的 2×2 压缩实验（a抽取式/b question-aware/c结合）全部跑完，**三版无一超过v10(直接喂,76%)**，v13c线上仅66.41（比v10低4.5分）。停下来回答用户三个尖锐问题——(1)问题1"agent推理"到现在解决了吗？(2)现在的流程算agent思考吗？(3)B榜没题、agent必须泛化"训练好"该怎么办？三个问题指向同一个根因与同一个解法。

### 14.1 三个诊断（承接第十二节的问题1，如实复盘）

1. **问题1(agent推理)到v13为止一行没解决**：v12/v13全在折腾"压缩prompt怎么写"，是换汤——**流程骨架始终是单步直线**（读题→固定检索→压缩一次→答一次→完），没有迭代/自检/动态决策/工具循环。压缩那步是"模型在压"，不是"agent决策要不要压、压多狠"。
2. **压缩当前是"错的压法"**：v13实验澄清——**治多选漏选的是question-aware(逐选项宁多勿漏)，不是抽取式(逐字摘原句，单用反而有害:mcq/multi都掉)**。但即便最好的v13b/c，multi也只打平v10、总分仍低。更暴露tf判断题新病：**压缩后tf从16/20掉到13-14/20，掉的全是"且/而"连接的复合判断句**（fin_a_010"比亚迪研发↑且美的回购"、res_a_018、fc_a_010）——压缩删掉了复合句其中一半的证据→整句判错。**这与多选漏选是同一个病：压缩丢掉分散证据点。**（注：用户曾疑"是否同义识别没训练好"，但数据显示单事实tf压缩前后都稳、只有复合句翻，故主因是证据覆盖不是语义理解。）
3. **B榜泛化才是终极目标**：A榜100题固定→反复看错题调参**天然诱导过拟合这100题**；B榜换全新题、没得看没得调，上传的是**同一套agent代码**，必须自己面对没见过的题也好。**所以"训练好agent"=让能力泛化、不依赖调过这100题**。继续抠TOP_N=针对100题调参→B榜失效；而"检索→答→自检→补查→修正"是**不依赖具体题目的通用推理能力**→换题照跑。**这就是为什么必须做agent推理循环，而不是继续调压缩prompt。**

### 14.2 参考文献（对口"给agent加推理循环"）

**① Self-RAG（arXiv:2310.11511，自反思检索增强）** — 治问题1
- 核心=**让模型边生成边自检**，四动作正好补上缺的推理循环：
  - **retrieve-on-demand(按需检索)**：不固定喂top-N，模型自己判断"要不要再查"→对应缺的"动态决策"
  - **自我批判relevance**：检索回来先问"这些段和题相关吗"
  - **自我批判support**：答完先问"我这答案被证据支持吗"→对应缺的"自检"
  - **反思循环**：生成↔反思交替，不合格就调整
- 这是问题1的标准解法：把"检索→答"直线，变成"检索→答→自检→不够就再查→修正"的循环。

**② CRAG（arXiv:2401.15884，纠错检索增强）** — 治压缩丢证据+检索不准
- **检索评估器**：先给检索质量打信心分(correct/incorrect/ambiguous)，不合格不硬喂。
- **decompose-then-recompose(拆解-重组)**：文档切细片→只留相关→重组，这正是**合规的抽取式压缩**，比v13多了"评估+纠错"。

**三个问题一个解法**：Self-RAG自检发现"某选项/某半句没证据"→CRAG式触发定向补查（在给定文档内BM25再查漏的那半段）→修正。**同时治压缩病(补回删掉的分散证据)、是真agent推理、且不依赖具体题目→泛化到B榜。**

### 14.3 ⚠️ 合规红线（思想可借，原版实现踩线）

- Self-RAG原版**训练模型**发射反思token→我们**不训练**，改**纯prompt让Qwen自检**（"每个选项都有证据吗？复合判断句每一半都验证了吗？"）→合规
- CRAG原版评估器是**小模型**+**web搜索兜底**→都违规（非Qwen、外部检索）。只借"评估→纠错"思想，用**Qwen自己判信心+在给定文档内BM25补查**，不联网、不用小模型→合规
- 全程**无embedding、无向量**→合规
- **结论：借两篇思想（自检循环+评估纠错），纯Qwen+BM25合规实现。除开发期模型例外，不踩红线。**

### 14.4 v14落地方向（真正的agent循环）

把单步管道升级成会推理的agent，核心循环：
```
检索(BM25) → question-aware压缩 → 答一次 → 【自检】"多选是否可能漏?tf复合句每一半都有证据吗?"
  → 若自检发现缺口:带着"缺哪个选项/哪半句"做定向补查(BM25再检索该关键词) → 补充证据 → 再答 → 收敛
  → 设最大轮数(防死循环)，收敛或到上限即定答
```
- 压缩保留v13b的question-aware（治多选最好那版）；重点是**加上答后自检+定向补查**。
- 单变量：相对v13b只加"自检→补查→修正"循环这一层。
- 开发期deepseek网关；跑完过`my_benchmark.py`看封存堆估分+过拟合仪表（这版尤其看**过拟合仪表**，因为目标是泛化）。

## 十五、agent循环全链路复盘：v15→v16→v17，与"必须做agent不是解题"的再定位（2026-07-14）

### 15.1 三版进程（一句话各自定位）

| 版本 | 变量 | 本地准确率 | token | 实测在线 | 结论 |
|---|---|---|---|---|---|
| v15 | agent循环搬到"直接喂"地基(去LLM压缩层) | 73%(tf 85%) | 370万 | 未提交 | 去压缩层有效，但循环重复喂全段→token爆炸倒亏 |
| v16 | v15循环换到强模型qwen3.7-plus | **77%**(multi **80%**) | 522万 | 未提交 | 准确率天花板确认77%，但token超预算触底 |
| v17 | v16+增量喂(合并自检+再答只喂增量) | 77%(tf **95%**) | **136万**(砍74%) | **68.8705** | token问题解决，但实测暴露本地虚高+agent没超过v10 |

主线：**先把准确率做到77%(v15/v16证明"直接喂+循环"能追平v10天花板)，再把token从522万砍到136万(v17增量喂)**。工程上全部达成，但**实测68.8705没突破v10的70.91**。

### 15.2 省token/找怀疑点的4个客观信号——全被数据否掉

为了给循环"按需触发"找一个客观、合规的怀疑信号（替代"模型自报missing"），做了4个零成本验证，全部否定：
1. **BM25置信度门控**——高/低批准确率仅差+10点、三分位不单调、mcq上信号反了。弱
2. **按题型砍mcq循环**——mcq循环净有益(2题改动里1题救回、0题改坏)，不能砍
3. **BM25反向识别漏选**——漏选vs真不选均分仅差+22、大面积重叠、精确率15~22%。无效
4. **logprobs捕捉漏选犹豫**——单字母无区分度；multi测试**意外反转**：发现v15漏选很大程度是**网关deepseek模型弱**造成的(6/10换qwen-plus直接答对)，不全是结构病。真硬漏选漏字母概率0.03~0.22(低概率确实=该补查)，但那模型额度不够跑全量
- **绕一圈的结论**：合规工具箱(纯prompt+BM25，无小模型/训练/联网)里，**"模型自己自检自报missing"可能已是最合理的怀疑点判断**。别的客观信号要么太弱要么违规

### 15.3 ★本session最重要的方法论教训：本地估分系统性偏乐观~2点

- v17本地估70.71、实测**68.8705**，差约2分。反推真实准确率≈**75%**(本地77%虚高约2点)
- 根因：本地"Claude尺子"(reference_claude.csv)**不是官方答案**。本地77%里有约2点是"跟Claude一致但其实错"
- **纪律更新：以后所有版本的本地分都要打~2点折扣看，不能拿本地估分当在线分的替身**

### 15.4 历次提交实测分反推真实准确率（把token项剥离）

用`实测分/(0.7+0.3×TokenScore)≈准确率`反推：

| 提交 | 实测分 | 反推真实准确率 | 说明 |
|---|---|---|---|
| v9(deepseek网关) | 76.4512 | **≈83%** | 真实天花板，但网关模型不可提交 |
| v10(qwen3.7-plus直接喂) | 70.9071 | ≈77% | 合规最强基线，**至今最高实测分** |
| v17(qwen3.7-plus+循环+增量喂) | 68.8705 | ≈75% | 我们的agent，没超过v10 |
| v13c | 66.4077 | ≈73% | 压缩版 |
| v14(压缩+循环) | 63.2266 | ≈71% | 循环套在压缩地基上 |

### 15.5 诚实存疑：不能下"循环伤准确率"的结论

v17(75%) vs v10(77%)差2点，但**不能归因于"循环伤准确率"**：
1. v17相对v10改了**两个变量**(加循环 + 合并自检prompt)，不是单变量对照
2. 2点在尺子噪声内
3. **v16纯循环从未上线**，"强模型+纯循环"的真实在线效果**未验证**
- 所以"循环在强模型上伤准确率"是**未证实的猜测**。要下结论必须先有单变量的在线对照(v16纯循环 vs v10)

### 15.6 战略再定位：必须做agent，不是解题；还有B榜

用户校正的核心方向（覆盖我之前"砍循环换模型刷A榜分"的错误建议）：
- **比赛名字就叫"金融长文本agent的压缩与上下文能力"——必须做的是agent，不是解题(解题=针对A榜100题调参刷分)**
- **还有B榜(未知题)**：任何针对A榜过拟合的技巧到B榜会失效，**泛化能力才是终极目标**
- **增量喂(v17)正是比赛要考的"动态记忆压缩"能力**——这个方向本身就对，是agent的核心能力，不是要砍掉的东西
- **真正的问题不是"该不该有循环"(agent必须有)，而是"我们的agent还不够好"**：循环的"再答"有时把对的初答改错(强模型过度自我怀疑，本该只补漏却删改了对的初选)

### 15.7 下一步方向（待用户确认，未开始写码）

- **路线A(我倾向)：让agent动作更保守**——多选题的"再答"只在新证据明确支持下**增选**，**绝不删改初答已选的选项**；tf/mcq同理只在补查带来新证据时才改。目的=把循环从"净持平/微亏"做成"只增益不倒退"，既保住agent形态又不让它把对的改错
- **路线B：先把v16纯循环上线**，拿到"强模型+纯循环"的干净在线分，与v10单变量对照，彻底搞清"循环到底帮还是伤"再决定
- 等用户在A/B之间选定后再动手（遵守"没确认第x步做好前不开始x+1步"）

### 16.3 论文/仓库完整清单(去重)

- **BM25** — Robertson & Walker 1994, Okapi。检索地基
- **RECOMP** — arXiv:2310.04408《Compressing Retrieved Documents for RAG》。压缩抽取式vs生成式、为答对优化
- **LLMLingua / LongLLMLingua** — 微软开源(GitHub, MIT)。question-aware压缩、coarse-to-fine。**库违规(小模型)，仅借思想**
- **Self-RAG** — arXiv:2310.11511《Self-Reflective RAG》。自检relevance/support、retrieve-on-demand
- **CRAG** — arXiv:2401.15884《Corrective RAG》。检索评估器+纠错式补查
- **Adaptive-RAG** — arXiv:2403.14403。按查询复杂度路由检索
- **FLARE** — arXiv:2305.06983《Forward-Looking Active REtrieval》。低置信token触发按需检索
- **DRAGIN** — arXiv:2403.10081。RIND实时信息需求检测+QFS查询构造

### 16.4 一句话血统总结

**检索地基=BM25；压缩层=RECOMP+LLMLingua(抽取式/question-aware/为答对优化)；agent循环=Self-RAG+CRAG(自检+纠错补查)；按需/增量=Adaptive-RAG+FLARE+DRAGIN(按需触发、只喂增量)；保守动作=自研(把"病=漏选"编进动作空间)。所有原版实现凡踩 小模型/训练/embedding/联网/非Qwen 红线的，一律只借观点、用"纯Qwen+纯BM25+纯逻辑"重写。**
