# 2026-06-29 Two Track Strategy

## Date

2026-06-29

## Context

目前已经完成 `ins_a_007` 的阶段 00 学习闭环：从人工读题、工具读取题目、doc_id 映射、人工查证据，到整理人工判断记录和 research log。

新的问题是：如果只沿着学习线慢慢推进，可能长期没有真实线上反馈；但如果直接冲 baseline 和提交，又容易变成不理解系统、只凭感觉乱改。

因此建立“双线推进”策略。

## Position In Roadmap

这不是替代原学习路线，而是在原路线旁边增加一条提交反馈线。

```text
学习理解线：人肉解题 -> 工具辅助 -> 模型证据判断 -> 自动跑题 -> 理解 baseline
提交反馈线：跑通 baseline/最小系统 -> 生成提交 -> 上传 -> 获得反馈 -> 记录失败类型
```

最终目标不是让两条线长期分裂，而是逐步交融。

## Goal

建立两条线的边界和协作方式：

- 学习理解线负责建立判断力。
- 提交反馈线负责尽快获得真实比赛反馈。
- 两线交融点负责把理解变成可提交系统，把反馈变成下一步学习重点。

### 学习理解线

```text
workspace/01_agent_learning/
```

目标：

```text
慢慢理解人怎么解题、工具怎么辅助、Agent 怎么循环，以及 baseline 每个模块为什么存在。
```

当前状态：

```text
下一步是工具辅助 1 题，例如补充 doc_id 映射工具和关键词搜索工具。
```

节奏：

```text
慢，重理解，重可解释，重记录。
```

### 提交反馈线

位置：

```text
workspace/03_baseline_improvement/submissions/
```

目标：

```text
后续尽快跑通一次 baseline 或最小提交流程，生成提交文件，上传并获得线上反馈。
```

第一次提交的目标不是高分，而是：

- 确认提交格式是否正确。
- 知道线上反馈如何呈现。
- 记录 baseline 或最小系统的失败类型。
- 建立后续实验对比基线。

节奏：

```text
快，重跑通，重反馈，重失败记录。
```

## Boundaries

两条线一开始必须分开，避免混乱：

- 学习理解线不被提交压力打乱。
- 提交反馈线不被过度研究拖慢。
- 学习脚本先放在 session 内，不直接污染正式工具。
- 提交反馈线的输出、提交记录和失败分析放在 `submissions/` 或相关实验目录。

## Merge Points

两条线最终要逐步交融：

```text
学习线产出的工具、关键词策略、证据判断方法
-> 进入提交反馈线，改进系统

提交反馈线得到的错题、格式错误、失败类型
-> 回到学习线，决定下一轮重点学习什么
```

典型交融点：

- `inspect_question.py` 这类学习工具被验证后，可以提升为通用工具。
- 人工解题记录中的关键词和判断点，可以变成检索规则或 prompt 结构。
- 线上错题可以回到 `manual_solving/` 做单题复盘。
- 提交失败类型可以进入 `error_analysis/`，指导 baseline 优化。

## Research Notes

双线策略的核心不是“分裂成两个项目”，而是降低复杂度：

```text
先分线，是为了让学习和提交各自有清晰目标。
后交融，是为了把理解变成成绩，把反馈变成下一轮学习方向。
```

这个策略能避免两个极端：

- 只学习不提交，长期没有真实反馈。
- 只提交不理解，改动不可解释、不可复现。
