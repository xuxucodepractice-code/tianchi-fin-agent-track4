- 修订：2026-07-16；当前执行状态见 `ablation_progress.md`。方法规则依据 `audits/playbook_review_2026-07-13.md`。

## 1. 目的

本文件用于指导 A 榜提交线的后续优化。目标不是一次性堆叠多个改动，而是通过连续、可归因的单变量实验，定位以下链路中真正的失分源：

```text
原始文档
→ 文档解析
→ Chunk 生成
→ 检索召回
→ 证据组织
→ Qwen 推理
→ 答案合成
→ 提交产物
→ 线上评分
```

最终需要回答：

1. 错题主要产生在哪一层？
2. 哪个改动真正增加了正确题数？
3. 哪些改动只有工程改善，但没有穿透到最终答案？
4. 哪些改动应保留，哪些应回滚？

当前已知基准（分数 + 精确 Token 可反推精确正确题数，每次上线都是一次无噪声测量）：

| --- | ---: | ---: | ---: | --- |
| v0 | 63.2607 | 1,161,593 | 精确 68/100 | 不可变原始对照与回滚基准 |
| v1s1 | 61.8540 | 1,280,096（上传精确值） | 精确 67/100 | 混合多变量，净 -1 题 +118k Token；本地重建产物 1,243,448 为污染值，不作依据 |

## 2. 六条实验纪律

### 2.1 一次只改一个变量

允许：

```text
只把 TF 从逐选项判断改成题干级判断。
```

禁止：

```text
同时修改解析、检索、Prompt、fallback 和 top-k。
```

### 2.2 没有本地净收益，不上传

所有实验必须先经过：

```text
本地真值集 → 答案 diff → 人工审计 → Go/No-Go
```

### 2.3 所有答案变化必须逐题解释

每一道发生变化的题都必须记录：

```text
旧答案 → 新答案
为什么变化
必要证据是否齐全
人工判断新答案是否更可能正确
```

### 2.4 每次实验必须结束于明确决策

实验使用四种结论，区分“可以保留代码”和“可以晋升为最佳提交父版本”：

```text
KEEP_SCORE      已证明有正分数收益，保留并成为新父版本
KEEP_CODE_ONLY  工程上可保留，但默认关闭，不进入 S10 集成
ROLLBACK        回滚，后续不继承
PILOT           样本不足，扩大标定后再决定
```

### 2.5 净收益必须高于重复噪声

qwen_client 已固定 temperature=0，但 API 仍可能非确定。对每种题型的首个试点，在同一题集、同一配置下重复运行一次。定义 `C` 为两次运行之间答案发生变化的题数，并分别记录 `churn_tf`、`churn_mcq`、`churn_multi`。此后对应题型的 Go 门槛加严为：

```text
N - M > C
```

### 2.6 没有正期望，不消耗槽位

上线前必须用 1.1 的标尺算期望分差。期望 ≤ 0 的实验只能记为 `KEEP_CODE_ONLY` 或 `ROLLBACK`，不得晋升为提交父版本，也不得消耗槽位（S4 是第一个适用案例）。

### 2.7 新 API 推理必须通过 Agent Trace Gate

从 E002 起，任何可能晋升为提交候选的新 API 推理必须满足：

```text
无答案 selection 先冻结
→ 空输出、空 cache fresh run
→ 完整 calls/messages/evidence/raw response/derivation trace
→ trace validator PASS
→ 候选先冻结并外部锚定 SHA256
→ 标签后揭晓并登记
```

已知标签题必须与 prospective blind 题分组。Gate 通过只证明链路可追溯，不能替代
`N-M>C`。当前 Gate 是本机自证层，不是 OS 级或第三方不可篡改证明；完整规则见
`governance/agent_trace_gate.md`。

## 3. 统一错误定位方法

每道错题按以下顺序检查，并标记一个主错误层。

| 检查 | 现象 | 主错误层 |
| --- | --- | --- |
| 原始文档是否包含答案 | 原文不存在或题目有歧义 | 数据/题目 |
| Chunk 是否包含关键原文 | PDF 有、Chunk 无 | 解析 |
| Top-k 是否召回关键 Chunk | Chunk 有、Top-k 无 | 检索 |
| 证据是否完整且主体正确 | Top-k 有，但公式断裂、主体错配或多文档缺失 | 证据组织 |
| Qwen 是否基于完整证据答对 | 证据完整、模型仍答错 | 推理/Prompt |
| 判断是否正确映射为答案 | 理由正确、最终字母错误 | 答案合成 |
| 三份提交产物是否一致 | 文件来自不同运行或缓存 | 提交流程 |

每题记录格式：

```yaml
qid: ins_a_001
gold_answer: B
current_answer: A
primary_failure: reasoning
secondary_failure: retrieval
required_facts:
  - 四款产品的身故保险金公式
  - 四个结果的确定性计算
  - 四个结果的排序
evidence_location:
  - doc_id/page/chunk_id
notes: 当前逐选项链路全部 insufficient，随后 fallback A
```

## 4. 每个实验的标准生命周期

### 4.1 Define：定义实验

实验开始前必须写清楚：

1. 认为哪一层有问题？
2. 支持这个判断的证据是什么？
3. 本次唯一修改变量是什么？
4. 哪些题预计被救回？
5. 哪些题存在回归风险？

### 4.2 Freeze：冻结父版本

记录：

- 父版本目录；
- SHA256；
- pipeline version；
- 100 题答案；
- Token；
- 线上分数；
- 缓存版本。

冻结的定义不是“文件存在”，而是**可运行状态**：以该父版本为底、rerun 集为空时，合并工具走空集快速路径，直接复制父版本三份原文件，输出 SHA256 必须与父版本完全一致。做不到就不允许以它为父版本开实验。

### 4.3 Offline：离线试验

先在本地真值集或证据覆盖集上运行，不直接全量提交。

对含新 API 推理的候选，“离线”不等于可以先看标签调答案。必须用冻结的无答案
selection 执行 fresh trace；生成代码不能读取盲标、历史答案或 Gold Oracle。运行后先验证
trace、合成并冻结候选，再首次揭晓 prospective 标签。

### 4.4 Diff：逐题比较

至少比较：

- 答案变化；
- 检索证据变化；
- 判断标签变化；
- rationale 变化；
- Token 变化；
- low confidence 变化。

### 4.5 Audit：人工审计

人工核对所有发生变化的标定题，计算：

```text
N = 旧错新对
M = 旧对新错
净收益 = N - M
```

判定门槛按 2.5 使用 `N - M > churn`。

### 4.6 Submit：线上提交

只有 `N - M > churn`、期望分差为正（按 1.1 标尺）、并通过全部护栏，才允许生成正式提交件。提交时记录当日槽位使用情况。

### 4.7 Decide：保留或回滚

线上结果出来后，记录：

- 与直接父版本的分差；
- Token 系数变化；
- 推算正确题数变化；
- KEEP_SCORE / KEEP_CODE_ONLY / ROLLBACK / PILOT。

### 提交前置门槛（先于一切执行）

```text
1. 先隐藏 v2 答案，独立盲标 reg_a_010、res_a_013，再揭晓两道翻转结果。
2. 两题都为 B → 净 +2 题，允许合成提交，预计约 65.0912 分。
3. 一题 B、一题 A → 一题改对、一题改错，净 0 题；Token 更高，预计约 63.2314 分，不提交。
4. 两题都为 A → 两题均从对改错，净 -2 题，预计约 61.3717 分，不提交。
5. 第 3、4 种情况最多记为 KEEP_CODE_ONLY（默认关闭），不得成为 S10 的提交父版本。
```
