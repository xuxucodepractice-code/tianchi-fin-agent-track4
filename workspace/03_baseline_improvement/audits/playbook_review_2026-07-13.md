## 一、总体结论

> **勘误（2026-07-13）**：本报告原先对 T1 两道 A→B 翻转题的收益计算有误。若两题中真值为 B 的数量为 `k`，净正确数变化为 `2k-2`：0 道 B 为 -2 题，1 道 B 为 0 题，2 道 B 为 +2 题。对应合成 Token 1,168,763 时约为 61.3717、63.2314、65.0912。只有两题都为 B 才有正收益；“≥1 题为 B 即提交”和“全 A 时 N=M=0”均已在主手册中修正。TF 标定也已改为预注册清单盲标，禁止以“找到 B”为停止条件。

方法论骨架是对的，可以直接沿用：单变量纪律、冻结父版本、N−M 逐题审计、Gold Oracle 三层定位（O1/O2/O3）、错误地图驱动优先级、KEEP/ROLLBACK/PILOT 三态决策，这套结构比 v1s1 时期的做法有质的提升，和 governance 文档也基本自洽。

但按现状执行会踩四个坑，按严重度排序：

1. **手册没有日历，而比赛还剩 8 天 A 榜 + 2.7 天 B 榜**。S0–S10 全流程装不进剩余时间，且 B 榜（决定最终名次）只占了一节两句话。
2. **"冻结 v0"冻结的是文件，不是可运行状态**。当前缓存已是 v0/v1s1/v2s1 三源混合，其中 ins_a_001/ins_a_002 还是 0-token 的 dry_run_mock。今天直接 `--all --resume` 会产出一个四源混合的怪版本。手册 S1 的完成门槛没有覆盖这一点。
3. **S4（TF 实验）的期望收益没有校准**。已核实：v2s1 的 TF 重跑缓存里只有 2 题翻转（reg_a_010、res_a_013），且 token 不降反升 +7,170。若这 2 题真值都是 A，提交 T1 会略微降分。
4. **没有提交次数账本**。每天 3 次，A 榜剩约 25 个槽位，B 榜只有约 8–9 个，手册各阶段都默认"随时可提交"。

## 三、评分模型精确换算（所有 Go/No-Go 的统一标尺）

公式：`Score = Correct × (0.7 + 0.3 × (5M − Tokens)/5M)`。在当前 token 水位（~1.16M）：

- **1 道题 = +0.93 分**
- **100k token = 0.41 分**，即 **1 道题 ≈ 228k token 等价**
- 分数 + 精确 token → 可**精确反推整数正确题数**。每次上线都是一次无噪声测量，这是手册"推算正确题数"一栏的理论保证，建议在手册第 1 节明确写出换算标尺，让每个实验的 Go/No-Go 都能折算成分数。

T1（TF 实验）的三种结局（以 v0 为父版本合成，token=1,168,763）：

| 情形 | 正确题数 | 线上分 | 相对 v0 |
| --- | ---: | ---: | ---: |
| 2 题翻转全错（真值均 A） | 68 | 63.2314 | **−0.03** |
| 翻转对 1 题 | 69 | 64.1613 | +0.90 |
| 翻转全对 | 70 | 65.0912 | +1.83 |

## 四、P0 疏漏（不修会直接损失分数或时间）

### 1. 无日历、B 榜位置错误

- 手册 18 节把 B 榜实验线放在总清单最后一项，无启动日期、无验收标准。但最终排名由 B 榜决定，B 榜窗口只有 2.7 天、约 8–9 个提交槽，且官方 baseline 显示无 doc_ids 的文档召回是最大难点（13/100 vs A 榜 49/100）。B 管线不可能在 7-22 之后现做。
- 你自己 7-08 的 v2 计划写的是"7-14 后启动 B 榜文档卡片"——明天就是 7-14，手册反而把它推后了。
- **修改建议**：§16 升级为带日期的并行线：≤7-16 启动文档卡片 BM25 选档开发；验收沿用 v2-4 的定义（A 榜 100 题隐藏 doc_ids 模拟，真 doc_ids ⊆ 选出集合的命中率 ≥90%，Recall@K 曲线定 K）；7-21 前完成 B 模式 dry-run；7-22 上午用第 1 个槽位提交冒烟版本，先验格式再迭代。注意 doc_id 命名空间异构（纯数字 / csrc_* / strict_v3_* / pack2_text*），卡片索引必须统一处理。
- 同时给 S7（R4/R5）、S8、S9 标注"时间不足时放弃"——8 天内大概率装不下，这三个阶段本来就设了错误地图门槛，再加一个日历门槛。

### 2. "父版本可运行状态"缺失（手册最危险的隐含假设）

- S4/S5/S6 都假设"其余 N 题从直接父版本复用"。但复用走的是 `processed_data/reasoning_samples/` 平铺缓存，**同一 qid 只有一份，版本互相覆盖**——这正是 v0 样本被 v1s1/v2s1 逐步吃掉、ins_a_001/002 变成 mock 的机制。archive_v0 只有 42/100，纯缓存已无法重建 v0。
- 后果示例：今天直接跑 `--all --use-qwen --resume` 会得到 60 题 v0 + 19 题 v1s1（含 8 道保险 multi 翻转，净 −1）+ 20 题 v2s1 TF + 2 题现管线重跑 —— 四源混合，分差无法归因到任何变量。
- **修改建议**（比修缓存更干净）：做**产物级合并**——实验提交 = 冻结父版本的 answer.csv/evidence.json（按 qid 取 80 题）+ 实验重跑产物（20 题）拼装。父版本从此是不可变文件而非可变缓存。S1 的完成门槛相应改为：**"rerun 集为空时，合并产物与 v0 冻结产物逐字节一致"**——这才证明父版本真正可复用。缓存目录改为按 pipeline_version 分目录（手册 S0 已列此项，落实即可），并清理 mock 样本（ins_a_001/002、q2.json）。
- 连带修一处规则矛盾：S0 要求 validator 检查"pipeline version 一致"，但 S4 的混合运行天然多版本。应改为 **lineage 声明式校验**：manifest 声明 parent_version + rerun_qids，validator 逐记录核对"rerun 集内 = 实验版本，rerun 集外 = 与父版本冻结产物完全一致（答案 + token 逐字段）"。这比"一致性"更强也更可执行。

### 3. S4（TF）期望收益未校准，存在负期望提交风险

- 已核实三件事：(a) 11 道已标定 TF 真值全为 A，v0 TF 本就 20/20 全 A；(b) v2s1 题干级链路在 20 题上只翻转 2 题（reg_a_010、res_a_013），另有 7 题 uncertain→fallback A；(c) token +7,170，**没有省 token 的红利**。
- 因此 T1 的全部上行空间 = 这 2 题的真值。手册把它们列入"特别检查"，方向对，但应升级为**提交前置门槛**：先人工标定这 2 题；≥1 题真值为 B 才提交；两题都是 A 则 T1 只做离线 KEEP（保留链路自洽性收益），不消耗槽位。同时预注册 N=M=0 时的决策（按手册规则是 PILOT，但在全 A 真值集上 N=M=0 是大概率事件，应提前写明走向）。
- S2 的 TF 补标应明确"**必须找 B**"：对 12 道未标定 TF 优先粗标，统计"TF 全 A 到底丢几题"——这是 v2-0c 里已有的关键洞察，手册丢掉了。

### 4. 提交预算无账本

- A 榜剩约 25 槽（今天起算），手册各实验合计需 8–11 次上线 + B 榜需 6–9 次。总量够，但需要防两件事：单日超 3 次的排队冲突、最后一天（7-21）无缓冲。
- **修改建议**：§17 模板加两个字段："本次为当日第几次提交 / 全程第几次"；§18 总清单加"7-21 保留 ≥1 个槽位给 S10 集成版，7-20 为集成回归日"。

## 五、P1 疏漏（影响优先级判断与实验效率）

### 5. 错误质量分布可以先估出来，Multi 才是误差主体

32 道错题的题型分布可先验估算：TF 错 0–4（全 A 真值下 v0 可能全对）；MCQ 错 3–7（15 题 A=11 的分布 + 0-support fallback 缺陷 + ins_a_001 已证错）；**Multi 错 21–29，占误差 2/3 以上**。两个推论：

- **MCQ 是收益密度最高的实验**：题少（15）、缺陷已被审计证实、ins_a_001 有完整验收链条。S5 不依赖 Multi 标定，可以在 S2 完成 15 道 MCQ 标定后立即启动，不必等 40 题全齐。
- Multi 整题制意味着"改一个选项通常只是换一种错法"（v1s1 已付学费：8 题翻转净 −1）。S6 的试点设计是对的，但错误地图（S3）应对 Multi 加大采样，否则 15/65 的标定密度不足以支撑 U1 的 Go/No-Go 外推。

### 6. S2 的 40 题全深度标定装不进日历

每题要求"全部事实点 + 公式 + doc/page/chunk + 三版本答案 + 双层错误归因"，按 v1 审计的实际耗时，40 题约需 2–3 天，会吃掉实验窗口。**建议分级**：Tier-1（真值 + 1–2 条关键证据指针）覆盖 40 题；Tier-2（完整 evidence_requirements 卡片）只做进入 Gold Oracle 的 ~15 题。cases/README 的首批 5 题清单与此兼容。

### 7. 可重复性基线（churn）未定义

qwen_client 默认 temperature=0（好），但 API 仍可能有非确定性。小样本 N−M 若与随机漂移同量级则结论无效。**建议**：在任一试点集上同配置重跑一次，测得"零变量翻转率"，并把所有 Go 门槛改为 `N − M > churn`。一次性成本 ~10–15 万 token，一劳永逸。

### 8. 手册 §5 目录树与 governance/artifact_rules.md 不一致

§5 写 `outputs/{tests,experiments,submission}`，governance 与实际脚手架是 `outputs/{tests,dry_runs,experiments,candidates}` + 根目录 `submission/` 作唯一上传出口。按 governance 版本统一手册（根 submission/ 已被 paths.py 和全部工具链假定）。

### 9. S5 M1 规格含糊

"四选项一次性比较"未写明逐选项判断是**删除**还是**降级为参考信号**（v2 计划取后者）。两者 token 相差近一倍、evidence.json 结构不同，会影响 diff 与验收判读。需在实验定义时二选一写死，否则"唯一变量"本身就不唯一。同理 T1 的实现里其实带了"uncertain 重问一次"，严格说是两个变量，记录时应注明（现状已实现，不必拆，但归因时要意识到）。
