## 0. 一个先于所有分析的关键数字

评分公式反推：TokenScore = (5,000,000 − 1,161,593) / 5,000,000 = 0.768，系数 = 0.7 + 0.3 × 0.768 = 0.930。

**实际准确率 ≈ 63.2607 / 0.930 ≈ 68%，即约错 32 题。**

Token 只用了预算的 23%。当前每答对 1 题值约 0.93 分；而多花 100 万 token 只损失约 4.1 分系数分。**结论：现在是"准确率换分"阶段，不是"省 token"阶段。** 只要一项改动每 100 万 token 能多对 5 题以上就是净赚，这个门槛非常低。

## 1. 总体结论

- v0 架构（解析→chunk→BM25→逐选项判断→normalize）作为骨架是健康的：可复现、可校验、有缓存、有低置信标记，68% 已远超官方基线 49%。
- **最大的单点失分源是 tf 题：20 道 tf 答案全部是 A**。逐选项判断框架对 tf 结构性失效（选项文本是"正确/错误"元标签，不是可检索、可核查的陈述）。若真值近半为 B，这一项就丢约 8-10 题 ≈ 8-9 分。
- 第二失分源是**多实体计算题**（以 insurance mcq 为典型）：5 道 insurance mcq 全部 4×insufficient → fallback A 纯猜。这类题需要"按产品逐一取条款 + 数值计算"，单轮 top-5 检索给不出完整证据。
- 第三风险是 **multi 过度 support**：65 道 multi 中 44 道答了 3 个及以上选项（38 题答 3 个、6 题全选）。support 判准偏宽（"部分匹配不等于 support"约束执行不严），一个误 support 就错整题。
- 检索层的真实问题不是"召回不够"而是**"证据断裂 + 主体错配"**：25% 的证据被截断到 1200 字；ins_a_002 选项 A 的 top1 证据是"重大疾病列表"（错 section）；多产品题证据没有按产品分桶。
- 逐选项独立调用对 tf 浪费一倍 token（A/B 是同一命题的正反面），对 multi 则缺整题一致性校验。
- low_confidence=17 是个好雷达，但它**低估了风险**：multi 的 lc 恒为 0（只要有 1 个 support 就不触发），而 multi 才是 65 题的大盘。
- Codex 的任务拆分让工程质量很高（测试、校验、冻结、缓存都在），但把"逐选项判断 + 统一 top_k + 单轮检索"过早定成了架构常量，压缩了题型差异化的设计空间。
- B 榜迁移只有占位（domain 全域 fallback），候选文档定位完全未验证——这是 7-22 之后的生死项，但不是本周的事。

## 2. 当前系统主要瓶颈（按预期失分排序）

1. **tf 判定框架失效（≈8-10 题）**。证据：tf 20/20 答 A；9 题 lc（双 support / 双 insufficient / 双 refute 全 fallback 到 A）；11 题"正常"判定也全为 A。原因有二：(a) "正确/错误"是停用词，选项检索词退化为题干词，A/B 拿到相同证据、问两个对称问题，模型有顺从偏差（确认题干）；(b) fallback 规则"无法确定→A"把所有分歧都推向 A。tf 应该改为：对题干陈述本身做一次"true/false/uncertain"判定。
2. **多实体/计算题证据组装失败（≈4-6 题）**。ins_a_001/002 型：题目涉及 3-4 个产品各自的公式和数值。单查询 top-5 无法覆盖"每个产品的对应条款"；证据里甚至混入无关 section。模型行为其实是对的（insufficient），错在检索和证据组织没按实体分桶，也没有第二轮补检索，最后 fallback 纯猜。
3. **multi 过度 support（估计 ≈5-8 题）**。support 202 次 / refute 79 / insufficient 79，multi 平均每题 support 2.7 个选项。research/regulatory 的 support% 达 61-64%。判准宽的根源：prompt 只有一句"部分匹配不等于 support"，没有要求逐要素核对（主体、数值、时间、条件缺一不可），也没有整题一致性复核。
4. **mcq fallback 纯猜（≈4-5 题）**。mcq 答案分布 A=11/15 严重偏斜，其中 6 题是 0-support fallback。mcq 正确做法是"四选一比较"，逐选项独立判断天然容易全 insufficient 或多 support（8/15 题 lc 即是证据）。
5. **证据上下文不足（放大 1-3 的错误率）**。evidence 给 Qwen 时只有 doc_id+页码，没有文档标题/产品名映射（doc_id "1" 对应哪款产品，模型只能从正文猜）；25% 证据被截断；相邻 chunk 不合并，条款跨 chunk 断裂。
6. Token 层面没有瓶颈，只有机会：1.16M/5M，人均 1.16 万 token/题，还有 3.8M 预算可以花在二轮检索和复核上。

## 4. 架构改进建议（分层）

**文档解析层（低优先级，先不动）**
- 现状可用。表格结构丢失和 csrc HTML 噪声是真实缺陷，但按失分排序不在前列。
- 唯一值得做的小事：建立 `doc_id → 文档标题/产品名` 元数据表（从每份文档首 chunk 提取），供证据组织层标注来源。成本极低，对多产品题收益直接。

**检索层**
- 保留 BM25 骨架，但从"每选项一次检索"升级为**多路检索**：题干一路、选项一路、实体/数值一路（产品名、公司名、年份、百分比强制命中），结果合并去重。
- **邻近扩展**：命中 chunk 的前后各 1 个 chunk（同 doc 同页或相邻页）自动带入，解决条款断裂。chunk_id 有序，实现成本低。
- **实体分桶**：题目/选项中出现多个产品名或公司名时，保证每个实体至少有 2 条来自其对应文档的证据（doc 标题元数据 + 按 doc 配额），治 ins_a_001 型失败。
- top_k：统一 5 不算错，但应改为"每选项预算制"——单实体题 5 条即可，多实体题按实体×2-3 条。
- 按领域微调查询词权重可以做，但收益估计小于上述结构性改动，往后排。

**证据组织层（高优先级）**
- evidence 头部加来源行：`【文档：<标题/产品名>（doc_id=X）第 N 页 章节 S】`。模型判"主体错配"全靠这个。
- 相邻 chunk 合并后再截断，截断上限从 1200 提到 1600-2000（token 有的是）。
- 表格类证据（含多个数字对齐的 chunk）不截断中部，保留整块。

**Prompt / 推理层（最高优先级）**
- **tf 专用**：一次调用，直接判"题干陈述为 真/假/无法确定"，证据为题干检索结果；fallback 不再恒 A（见答案合成层）。
- **mcq 专用**：一次调用给全部四个选项 + 各自证据，要求"比较后选唯一最优，输出选项与排序理由"。逐选项独立判断保留为参考信号，不再是唯一决策源。
- **multi 保留逐选项**，但 support 判准收紧：要求逐要素核对（主体/数值/日期/条件），任一要素证据缺失就不得 support；再加一次整题一致性复核调用（把四个判断+理由回给模型："这些判断互相矛盾吗？哪个最可疑？"）。
- 允许模型输出 `need_more_evidence` 信号 + 建议检索词，驱动二轮检索（限一轮，防失控）。
- 计算题：让 Qwen 在 rationale 中列出算式，代码侧用 Python 复核算术（这不是"非 Qwen 模型参与推理"，是确定性计算校验，合规）。

**答案合成层**
- mcq 多 support 时"按 evidence_refs 数量选"不可靠——改为触发二轮比较调用。
- tf fallback 恒 A 必须废除：无法确定时改为触发一次"直接判题干"重问；仍不确定才 fallback，且 fallback 方向可以用"insufficient 偏多 → 题干存疑 → B"这类启发（待实验一的人工核对结果标定）。
- mcq 0-support 时同样触发二轮（放宽证据或换检索词重试），而不是直接 A。
- multi 无变化（fallback 到 A 的 multi 为 0，规则未实际触发）。

**Token 策略**
- 预算再分配：基础轮维持 ~1.2M；二轮/复核预算 1.5M，封顶 3M 总量（系数 0.82，需 Acc>68.6% 即回本——只要净多对 1 题就打平，多对 3 题净赚 2 分以上）。
- 分配原则：low_confidence、0-support mcq、tf 不确定、multi 一致性存疑的题才进二轮；正常题不追加。
- 需要在 manifest 中加"每题二轮 token"字段，实验期监控。

**B 榜迁移能力**
- 当前只有 domain 全域 fallback，B 榜等于裸奔。但正确的时间点是 A 榜优化收敛后（约 7-14 之后）。
- 提前可做的低成本准备：给 68 个文档建"文档卡片"（标题、主体实体、年份、类型关键词），B 榜时先用 BM25 在卡片上选文档、再进正常管线。架构上 `filter_chunks_for_question` 已有插槽，改动面小。

## 5. 最优先的 5 个 v1 实验

**实验一：tf 判定框架重做**
- 目标：消除 tf 全 A 偏置，救回 tf 失分。
- 改动范围：prompts.py 加 tf 专用 prompt；reason_qwen.py 对 tf 走单次"题干真/假/uncertain"判定；normalize_answer.py tf 分支改造。
- 预期收益：+4 至 +9 题（取决于真值中 B 的占比；先人工核 5 题标定）。全实验中性价比最高。
- 风险：低。tf 只有 20 题，影响面清晰可控。
- 验证：改后 tf 答案分布应不再全 A；对人工核过的 5 题逐一比对；线上提交对照分。
- 需要重新调用 Qwen：是，仅 tf 20 题（其余 80 题走 reasoning_samples 缓存复用，机制已具备）。
- 预计 token：~12 万（20 题 × 1 次调用，比 v0 的 2 次/题更省）。

**实验二：mcq 全题比较判定 + 0-support 二轮**
- 目标：救 6 道纯猜 mcq 和 2 道多 support mcq。
- 改动范围：prompts.py 加 mcq 比较 prompt；reason_qwen.py mcq 分支（保留逐选项判断作为输入信号）；normalize_answer.py mcq 决策规则。
- 预期收益：+3 至 +5 题。
- 风险：中。全题比较可能引入"模型偏好流畅选项"的新偏差；保留逐选项信号做交叉检查可缓解。
- 验证：mcq 答案分布不再 A=11/15；lc mcq 数下降；ins_a_001 等 5 题人工核对证据链。
- 需要重新调用 Qwen：是，仅 mcq 15 题。
- 预计 token：~25 万（15 题 × [4 次逐选项 + 1 次比较]，其中逐选项部分可复用 v0 缓存则减半）。

**实验三：证据组织升级（来源标注 + 邻近合并 + 实体分桶）**
- 目标：把 insurance insufficient 49% 压下来，给实验二的 mcq 提供可判的证据。
- 改动范围：retrieve.py（邻近扩展、实体配额）；新增 doc 标题元数据小模块；prompts.py 证据格式化。不动 parse_documents。
- 预期收益：+2 至 +4 题（主要在 insurance/fin 多实体题），并放大实验一/二效果。
- 风险：中。证据变长 → prompt token 上升约 40-60%；需监控截断率和 token。
- 验证：ins_a_001/002 的证据中每个产品至少 2 条来自对应文档；insufficient 率变化；离线可先验证（不调 Qwen，先看检索产物）。
- 需要重新调用 Qwen：检索部分不需要；效果确认需要重跑受影响题（约 30-40 题）。
- 预计 token：~60 万。

**实验四：multi support 判准收紧 + 整题一致性复核**
- 目标：压掉 multi 过度 support（44/65 答 ≥3 项，6 题全选）。
- 改动范围：prompts.py（逐要素核对清单）；reason_qwen.py 加一致性复核调用；normalize 采纳复核结果的规则。
- 预期收益：+2 至 +5 题；但也可能把判对的改错（见风险）。
- 风险：**本实验风险最高**。判准收紧会把部分正确 support 推成 insufficient；一致性复核可能引入犹豫。必须小样本先行。
- 验证：先只跑 multi 全选 6 题 + 3-support 中抽 10 题，人工比对新旧判断；确认净收益为正再全量。
- 需要重新调用 Qwen：是。
- 预计 token：试点 ~20 万，全量 multi 复核 ~80 万。

**实验五：低置信题定向二轮（need_more_evidence 循环）**
- 目标：给 lc/0-support/双不确定题一次带新检索词的补检索+重判机会。
- 改动范围：reason_qwen.py（模型可返回补检索建议词）；retrieve.py 支持追加查询；run_submission 二轮编排；manifest 记二轮 token。
- 预期收益：+1 至 +3 题（与实验一/二有重叠，收益是剩余部分）。
- 风险：中。循环必须硬性限一轮；防止模型永远说"不够"。
- 验证：二轮后 insufficient→support/refute 的转化率；转化题人工抽查是否真的判对了。
- 需要重新调用 Qwen：是，仅触发题（预计 15-25 题）。
- 预计 token：~30 万。

五个实验全部做完的 token 增量约 +230 万，总量 ~3.4M，系数降到 0.796；打平需要 Acc ≈ 79.5%（+11.5 题）。上述保守收益合计 +12 至 +26 题，中位估计能到 78-85%，对应 62-68 分的下限风险和 70+ 的合理预期。**因此建议不要五个全上，按顺序做一/二/三，视线上反馈再决定四/五**——一/二/三合计 token 增量只有 ~100 万（总 2.2M，系数 0.868，打平只需 +5 题，而预期是 +9 至 +18 题）。

## 6. 哪些事情暂时不要做

- **不要重做解析层**（换 PDF 解析器、上 MinerU、表格重建）：工程量大、影响面全局、当前失分主因不在这。等实验一/二/三消化后，若财报表格题仍是主要错误源再动。
- **不要现在优化 B 榜检索**：B 榜 7-22 才开始，现在做无法验证效果。只做低成本的"文档卡片"准备。
- **不要全局调 BM25 参数/领域权重**：微调收益小、评估噪声大（没有本地标准答案），容易陷入无法归因的调参循环。
- **不要上 agent 框架/多轮自主循环**：当前固定 workflow 可观察、可复现、可缓存复用；自主循环会破坏这三点，且 token 不可控。
- **不要为省 token 做证据压缩**：预算用了 23%，压缩是负优先级。
- **不要动 normalize 的 multi 排序去重逻辑和校验器**：这是格式合法性的生命线，v0 已验证 100 题零格式错。

## 7. 对 Codex Task 1-7 prompt 的反思

合理的拆分（应保留的做法）：
- 骨架先行（Task 1 先打通 answer.csv 全链路）、映射先于解析（Task 2）、每任务带测试和验收标准、产物冻结和 SHA256 基准、dry-run/正式模式分离、缓存复用（77/100 复用命中）。这些让 7 个任务零返工，工程质量是真实优势。

限制了系统能力的地方：
- **"逐选项判断"在 Task 5 prompt 里被规定为唯一推理形态**，并向下传导到 normalize 规则。tf 全 A 偏置就是这个设计常量的直接后果。题型差异（mcq 是比较题、tf 是命题判定题、multi 才是逐项核查题）在任务描述里被抹平了。
- **top_k=5、每选项统一预算**在 Task 4 被写死成验收标准，多实体题的证据需求差异没有表达空间。
- **单轮流水线假设**：Task 4→5 的接口是"检索结果一次性交给判断"，没给"判断后回头补检索"留接口，导致 insufficient 只能认命。
- **fallback 规则在 prompt 里被逐字规定**（"tf 无法确定返回 A"），这本该是需要数据标定的决策，却被当作格式要求固化。
- 每个 Task 的"限制要求"清单很长（不要做 X/Y/Z），执行安全性好，但也意味着执行者不会顺手修掉相邻层的明显问题（例如 Task 4 时就能看出 tf 选项无检索词，但越界修复不在授权内）。

后续 prompt 建议怎么写：
- 定义**目标和验收指标**（如"tf 答案分布不得单边>70%""insurance insufficient 率<30%"），把实现方式留给执行者。
- 用**不变量**代替实现细节清单：合规红线（Qwen-only、无 embedding）、格式校验、产物冻结、缓存兼容——这些必须逐字规定；其余（prompt 形态、检索轮数、top_k）交给实验验证。
- 每个任务留一节"执行者观察到的越界问题"，允许记录但不实施，避免问题被流程静默吞掉。
- 大改动要求先小样本试点 + 人工核对，再授权全量（本报告实验四即按此设计）。
